# lfguard 0.7.1

`lfguard 0.7.1` makes review outputs safer for advisory DMS and LLM workflows by
separating technical severity from user-facing workflow action.

## Advisory Review Actions

- Adds `recommended_action` to lint findings, audit findings, plan changes, and
  review grant-change evidence.
- Adds `hard_block` to the same records so integrations can distinguish
  "policy risk" from "do not proceed".
- Adds review summary `recommended_action`, `action_summary`, and
  `blocking_reasons` fields.
- Keeps existing `severity` values intact. `severity: error` means the policy is
  risky or invalid under lfguard rules; it no longer means the review bundle is
  automatically blocked.

## Blocking Semantics

- `lfguard review --fail-on-blocked` now fails only when review status is
  `blocked`, which is driven by hard-block findings or destructive planned
  changes.
- Broad principals, broad permissions, mutating grants, grantable permissions,
  and named grants keep `severity: error` but default to
  `recommended_action: approval_required`.
- Expired exceptions, duplicate identities, invalid/undefined policy references,
  AWS-invalid permission combinations, and destructive planned changes default
  to `recommended_action: block`.
