# Terraform and CDK Coexistence

Use Terraform, CloudFormation, or CDK for infrastructure provisioning. Use
`lfguard` for the Lake Formation permission layer when that layer needs review,
drift checks, explanation, and conservative apply.

## Ownership Split

| Surface | Prefer IaC | Prefer lfguard |
| --- | --- | --- |
| AWS accounts, VPCs, buckets, KMS keys | Yes | No |
| IAM roles and trust policy | Yes | No |
| Glue databases and tables | Usually | No, except as named resources in policy references |
| Registered data lake locations | Yes | No |
| `DATA_LOCATION_ACCESS` grants | Optional | Yes when tied to permission lifecycle |
| LF-Tag definitions and values | Optional | Yes when governed as policy |
| LF-Tag assignments | Avoid double ownership | Yes when access policy depends on them |
| Named LF-Tag expressions | Avoid double ownership | Yes when grants reference them |
| Lake Formation grants | Avoid double ownership | Yes when review and drift evidence matter |
| Data cells filters | Avoid double ownership | Yes when filtered access is governed as policy |

The key rule is simple: do not let two tools write the same Lake Formation
policy surface. If Terraform owns a grant, do not also put that grant in
`lfguard` desired state. If `lfguard` owns LF-Tag assignments, keep those
assignments out of Terraform.

## Common Pattern

1. Terraform or CDK creates IAM roles, S3 buckets, Glue databases, tables, and
   registered data lake locations.
2. The IaC pipeline exports stable identifiers such as role ARNs, catalog IDs,
   database names, table names, and data-location ARNs.
3. `policy.py` consumes those identifiers from checked-in constants or generated
   environment files.
4. CI runs `lfguard check`, `audit`, `plan`, and `explain` as policy evidence.
5. A separately scoped apply step runs `lfguard apply` only after review.

Example constants file generated by an IaC pipeline:

```python
CATALOG_ID = "111122223333"
ANALYST_ROLE = "arn:aws:iam::111122223333:role/FinanceAnalyst"
PRODUCER_ROLE = "arn:aws:iam::111122223333:role/FinanceProducer"
RAW_LOCATION = "arn:aws:s3:::finance-lake/raw/"
```

Then `policy.py` imports the constants and maps them into permission bundles:

```python
from lakeformation_guard.policy import LakePolicy, TagAssignmentScope, data_location_access, producer, reader
from iac_outputs import ANALYST_ROLE, CATALOG_ID, PRODUCER_ROLE, RAW_LOCATION

policy = LakePolicy()
policy.tag_key("domain", values=["finance"], assignable_to=[TagAssignmentScope.DATABASE, TagAssignmentScope.TABLE], catalog_id=CATALOG_ID)
policy.tag_database("finance_curated", domain="finance", catalog_id=CATALOG_ID)
policy.tag_table("finance_curated", "invoices", domain="finance", catalog_id=CATALOG_ID)

policy.group("finance_readers", reader(catalog_id=CATALOG_ID).where(domain="finance"))
policy.group("finance_producers", producer(catalog_id=CATALOG_ID).where(domain="finance"))
policy.group("finance_raw_location", data_location_access(RAW_LOCATION, catalog_id=CATALOG_ID))

policy.bind_role(ANALYST_ROLE, "finance_readers")
policy.bind_role(PRODUCER_ROLE, ["finance_producers", "finance_raw_location"])
```

## Drift Ownership

During adoption, current AWS state often contains grants that are intentionally
outside the new policy repository. Model that explicitly:

```json
{
  "ownership": {
    "managed_principals": ["arn:aws:iam::111122223333:role/data-*"],
    "unmanaged_action": "warn"
  },
  "ignore": {
    "principals": ["IAM_ALLOWED_PRINCIPALS"],
    "resources": [{"database": "legacy_*"}]
  }
}
```

Use warnings while migrating. Move to errors when the repository is the source
of truth for the managed surface.

## Terraform/CDK Guardrails

- Keep `lfguard apply --execute` out of the same job that runs Terraform apply
  unless both plans are reviewed together.
- Run `lfguard plan --fail-on-changes` after IaC changes that create new roles,
  tables, or data locations.
- Use read-only Lake Formation permissions for CI evidence jobs.
- Use a narrower additive apply role for routine changes.
- Keep destructive flags in a separate job with explicit approval.

## When Terraform Should Still Own Grants

Terraform or CDK can own simple static grants when the access pattern changes
rarely and review evidence is not needed beyond IaC plan output. `lfguard` is
more valuable when teams need LF-Tag policy validation, access explanation,
exception metadata, drift reports, selective apply, or safety budgets.
